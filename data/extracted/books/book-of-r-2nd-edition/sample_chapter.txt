4.1 Logical Values
        Logical values (which we also call logicals) are based on a simple premise:
        A logical­valued object can be either TRUE or FALSE. We interpret these as
        yes/no, one/zero, satisfied/not satisfied, and so on. This concept appears
        across all programming languages. Often, logical values signal whether a
        condition has been satisfied or whether a parameter should be switched on.
             You encountered logical values briefly when you used the sort function
        in Section 2.3.2 and the matrix function in Section 3.1. When using sort,
        setting decreasing=TRUE returns a vector ordered from largest to smallest,
        and decreasing=FALSE sorts the vector the other way around. Similarly, when
                 Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

                  constructing a matrix, byrow=TRUE fills the matrix entries row­wise; otherwise,
                  the matrix fills column­wise. Let’s take a more detailed look at ways to use
                  logicals.


                  4.1.1 TRUE or FALSE?
                  We write logical values in R as TRUE and FALSE, though we can abbreviate
                  them as T or F. The abbreviated version has no effect on the execution of the
                  code; for example, using decreasing=T is equivalent to decreasing=TRUE in the
                  sort function. (Note, however, that you shouldn’t name other objects T or F
                  in your workspace if you plan to use this convenience—see Section 9.1.3.)
                      Assigning logical values to an object is the same as assigning numeric
                  values:

                  R> foo <- TRUE
                  R> foo
                  [1] TRUE
                  R> bar <- F
                  R> bar
                  [1] FALSE

                      This gives you one object with the value TRUE and one with the value
                  FALSE. Similarly, you can fill vectors with logical values:

                  R> baz <- c(T,F,F,F,T,F,T,T,T,F,T,F)
                  R> baz
                  [1] TRUE FALSE FALSE FALSE TRUE FALSE TRUE TRUE TRUE FALSE TRUE FALSE
                  R> length(x=baz)
                  [1] 12

                      You can create matrices (and other higher­dimensional arrays) with
                  these values, too. Using foo and baz, you might construct something like this:

                  R> qux <- matrix(data=baz,nrow=3,ncol=4,byrow=foo)
                  R> qux
                       [,1] [,2] [,3] [,4]
                  [1,] TRUE FALSE FALSE FALSE
                  [2,] TRUE FALSE TRUE TRUE
                  [3,] TRUE FALSE TRUE FALSE

                  As expected given the use of byrow and the stored value of foo, the resulting
                  structure has been filled in a row­wise fashion.

 often represent logical values in datasets with TRUE as 1 and FALSE as 0. In fact,

                  4.1.5 Logical Subsetting and Extraction
                  You can also use logicals to extract and subset elements in vectors and other
                  objects, in the same way as with index vectors. Rather than entering explicit
                  indexes in the square brackets, you can supply logical flag vectors, where an
                  element is extracted if the corresponding entry in the flag vector is TRUE. As
                  such, logical flag vectors and the vector that’s being accessed should be the
                  same length (though recycling does occur for shorter flag vectors, as you’ll
                  see shortly).
                       At the beginning of Section 2.3.3 you defined a vector of length 10 as
                  follows:

                  R> myvec <- c(5,-2.3,4,4,4,6,8,10,40221,-8)

                      If you wanted to extract the two negative elements, you could either
                  enter myvec[c(2, 10)] or do the following using logical flags:

                  R> myvec[c(F,T,F,F,F,F,F,F,F,T)]
                  [1] -2.3 -8.0

                  This particular example may seem far too cumbersome for practical use.
                  The core behavior is incredibly powerful, however, when you implicitly cre­
                  ate the desired flag vector based on the desired condition (or several condi­
                  tions). Taking the current example, remember that you can use logicals to
                  find negative elements in myvec by applying the condition <0:

                  R> myvec<0
                  [1] FALSE TRUE FALSE FALSE FALSE FALSE FALSE FALSE FALSE TRUE

                  You can use this flag vector to subset myvec to get the same result as earlier.
                  As such, the code

                  R> myvec[myvec<0]
                  [1] -2.3 -8.0

 4.2.1 String Creation
 We indicate character strings with double quotation marks ("). To create a
 string, enter text between a pair of quotes:

 R> foo <- "This is a character string!"
 R> foo
 [1] "This is a character string!"
 R> length(x=foo)
 [1] 1

     R treats the string as a single entity. In other words, foo is a vector of
 length 1 because R counts only the total number of distinct strings rather
 than individual words or characters. To count the number of individual
 characters, use the nchar function. Here’s an example using foo:

 R> nchar(x=foo)
 [1] 27

      Almost any combination of characters, including numbers, can be a
 valid character string, as shown here:

 R> bar <- "23.3"
 R> bar
 [1] "23.3"

      In this form, the string has no numeric meaning, and R won’t treat it
 like the number 23.3. Attempting to multiply it by 2, for example, results in
 an error:

 R> bar*2
 Error in bar * 2 : non-numeric argument to binary operator

     This error occurs because * is expecting to operate on two numeric
 values (not one number and one string, which makes no sense).
     You can compare strings in several ways, the most common being a
 check for equality:

 R> "alpha"=="alpha"
 [1] TRUE
 R> "alpha"!="beta"
 [1] TRUE
 R> c("alpha","beta","gamma")=="beta"
 [1] FALSE TRUE FALSE

      Other relational operators work as you might expect. For example, R
 considers letters that come later in the alphabet to be greater than earlier
 letters, meaning it can determine whether one string of letters is greater
 than another with respect to alphabetical order:

 R> "alpha"<="beta"
 [1] TRUE
                                                                     Non-Numeric Values   73
                 Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

                  R> "gamma">"Alpha"
                  [1] TRUE

                       Furthermore, R treats uppercase letters as “greater” than lowercase
                  letters:

                  R> "Alpha">"alpha"
                  [1] TRUE
                  R> "beta">="bEtA"
                  [1] FALSE

                     You can use most symbols in a string. The following string is valid, for
                  example:

                  R> baz <- "&4 _ 3 **%.? $ymbolic non$en$e ,; "
                  R> baz
                  [1] "&4 _ 3 **%.? $ymbolic non$en$e ,; "

                      One important exception is the backslash \, or escape. Using a back­
                  slash within the quotation marks of a string initiates some simple control
                  over the printing or display of the string itself. You’ll see how this works in
                  Section 4.2.3. First, let’s look at two useful functions for combining strings.


                  4.2.2 Concatenation
                  There are two main functions we use to concatenate (or glue together) one or
                  more strings: cat and paste. The difference between the two lies in how their
                  contents are returned. The first function, cat, sends its output directly to the
                  console screen and doesn’t formally return anything. The paste function con­
                  catenates its contents and then returns the final character string as a usable
                  R object. This is useful when R needs to pass the result of a string concatena­
                  tion to another function or use it in some secondary way, as opposed to just
                  displaying it. Consider the following vector of character strings:

                  R> qux <- c("awesome","R","is")
                  R> length(x=qux)
                  [1] 3
                  R> qux
                  [1] "awesome" "R"       "is"

                  As with numbers and logicals, you can also store any number of strings in a
                  matrix or array structure.
                       When calling cat or paste, you pass arguments to the function in the
                  order you want them combined. The following lines show identical usage
                  yet different types of output from the two functions:

                  R> cat(qux[2],qux[3],"totally",qux[1],"!")
                  R is totally awesome !
                  R> paste(qux[2],qux[3],"totally",qux[1],"!")
                  [1] "R is totally awesome !"

Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

 Here, you’ve used the three elements of qux as well as two additional strings,
 "totally" and "!", to produce the final concatenated string. In the output,
 cat simply concatenates and prints the text to the screen. This means you
 can’t directly assign the result to a new variable and treat it as a character
 string. For paste, however, the [1] to the left of the output and the presence
 of the double quotes indicate the returned item is a vector containing a char­
 acter string, which you can assign to an object and use in other functions.
      These two functions have an optional argument, sep, that’s used as a sep­

 string, it places this string between all other strings you’ve provided to paste
 or cat. For example:

 R> paste(qux[2],qux[3],"totally",qux[1],"!",sep="---")

 R> paste(qux[2],qux[3],"totally",qux[1],"!",sep="")

 paste and cat. If the sep argument isn’t included, R inserts a space between
 strings by default. For example, using manual insertion of spaces where nec­

 R> cat("Do you think ",qux[2]," ",qux[3]," ",qux[1],"?",sep="")
 Do you think R is awesome?

                  4.2.4 Substrings and Matching
                  Pattern matching lets you inspect a given string to identify smaller “query”
                  strings within it. For the simplest form of pattern matching, use the grep
                  function to inspect a vector of character strings and return the indexes (if
                  any) of the elements that contain the query string. For example:

                  R> poetry <- c("Hey","diddle","diddle","the","cat","and","the","fiddle")
                  R> result1 <- grep(pattern="iddle",x=poetry)
                  R> result1
                  [1] 2 3 8

                  Here, you apply grep to the vector of character strings in the poetry object
                  from Section 4.2.2. Via the pattern argument, you’re asking for matches to
                  the query string "iddle". The result, stored as result1, is a numeric vector
                  containing 2, 3, and 8—correctly identifying the positions of those strings
                  in poetry that contain the pattern of interest. When such a search yields no
                  results, you see the following:

                  R> result2 <- grep(pattern="dog",x=poetry)
                  R> result2
                  integer(0)

                  The somewhat curious notation integer(0) simply denotes an “empty” vector,
                  evidenced by use of length on the object:

                  R> length(x=result2)
                  [1] 0

                  This is fitting, seeing as there are no strings in poetry containing the sub­
                  string "dog".
                       A related function is substr, which takes a target string x and returns the
                  substring that sits between two character positions, indicated with numbers

 Here, you extract the characters between (and including) positions 21 and

 R> substr(x=poetry,start=1,stop=3)
 [1] "Hey" "did" "did" "the" "cat" "and" "the" "fid"

4.3 Factors
        In this section, you’ll look at some simple functions related to creating,
        handling, and inspecting factors. Factors are R’s most natural way of rep­
        resenting data values which take on only one of a finite number of distinct
        categories, rather than belonging to a continuum. Categorical data can play
        an important role in data science, and you’ll look at factors again from a sta­
        tistical perspective in Chapter 13.


        4.3.1 Identifying Categories
        To see how factors work, we’ll start with a simple dataset. Suppose you find
        eight people and record their first name, sex, and birth month in Table 4­4.

        Table 4-4: An Example Dataset of
        Eight Individuals
         Person      Sex       Birth month
         Liz         Female    April
         Jolene      Female    January
         Susan       Female    December
         Boris       Male      September

         Simon       Male      July
         Amy         Female    June


            There’s really only one sensible way to represent the name of each per­
        son in R, which is as a vector of character strings:

        R> firstname <- c("Liz","Jolene","Susan","Boris","Rochelle","Tim","Simon",
                          "Amy")

             You have more flexibility when recording sex, however. Coding females
        as 0 and males as 1, a numeric option would be as follows:

        R> sex.num <- c(0,0,0,1,0,1,1,0)

        don’t need to remember the numeric code for each group, as shown here:

        R> sex.char <- c("female","female","female","male","female","male","male",

        name and their sex when stored as data. Where a person’s name is a unique

                  of categories, as a factor in R.

                  can’t fill matrices or multidimensional arrays using factor values; factors can
                  take only the form of vectors. To create a factor, use the function factor, as
                  in this example working with sex.num and sex.char:

                  (R’s internal representation of factor objects is a bit different as well). Sim­

                  with vectors, for example.

 4.3.2 Defining and Ordering Levels
 The sex factor from the previous section represents the simplest kind of
 factor variable; there are only two possible levels with no ordering, in that
 one level is not intuitively considered “higher than” or “following” the other.
 Here, you’ll look at factors with more than two levels, as well as defining and
 customizing levels in greater detail.
     To motivate things, we’ll focus on the month of birth (MOB) data for
 the eight individuals in Table 4­4, first storing the corresponding entries as a
 simple character vector:

 R> mob <- c("Apr","Jan","Dec","Sep","Nov","Jul","Jul","Jun")

     Like sex, it makes sense to treat MOB as a categorical variable. If we use
 factor in exactly the same way as before, we get something like this:

 R> mob.fac.basic <- factor(x=mob)
 R> mob.fac.basic
 [1] Apr Jan Dec Sep Nov Jul Jul Jun
 Levels: Apr Dec Jan Jul Jun Nov Sep

 There are two potential problems with this basic version of the MOB factor.
 First, not all possible categories are represented because the original mob
 character vector contains only seven unique months. Tabulating the data

                 Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

                  with the function table, we see two observations categorized as “July,” with
                  the remaining months represented only once:

                  R> table(mob.fac.basic)
                  mob.fac.basic
                  Apr Dec Jan Jul Jun Nov Sep
                    1   1 1     2 1     1   1

                  Second, the levels of the factor aren’t arranged in the “natural” calendar
                  order of months; the default behavior in R is to sort alphabetically.
                      While these two issues may seem minor in isolation, they can have
                  important consequences when wrangling data and using categorical vari­
                  ables in statistical analyses. You can deal with both problems to varying
                  extents by customizing additional arguments to the factor function. The
                  most important of these is the levels argument, which takes a character vec­
                  tor defining all the possible categories you want, in the order that you want.
                  Redefine the MOB factor as follows:

                  R> ms <- c("Jan","Feb","Mar","Apr","May","Jun","Jul","Aug","Sep","Oct","Nov",

                  Levels: Jan Feb Mar Apr May Jun Jul Aug Sep Oct Nov Dec

                  Here, R defines the mob.fac vector in terms of the same eight individual
                  entries (the mob object) as earlier. The new factor has 12 levels due to using
                  levels=ms inside the call to factor, even though you haven’t actually made
                  observations for the levels "Feb", "Mar", "May", "Aug", or "Oct". (Note that if
                  your R console window is too narrow to print all the levels to the screen,
                  you may see ... , indicating there’s more output. Widen your window and
                  reprint the object to see the hidden levels.) In addition, R arranges the new
                  category levels in the order you supplied them to the levels argument. Once
                  more using the table function, you obtain the following:

                  R> table(mob.fac)
                  mob.fac
                  Jan Feb Mar Apr May Jun Jul Aug Sep Oct Nov Dec
                    1   0 0     1 0     1   2 0     1   0 1     1

                  such as regression and other types of modeling: It changes R’s handling and interpre­
                  tation of categorical variables. As such, it’s used only in very specific circumstances

 The individual entries are entirely unchanged; only the order of the level
 definitions are modified.
      The relevel function is useful if you just want to ensure a particular level
 comes first (we sometimes refer to this as a reference category or reference level).
 As the sex factor has only two levels to begin with, you could execute the
 following line to equivalently create sex.char.fac2:

 R> relevel(x=sex.char.fac,ref="male")
 [1] female female female male female male           male    female
 Levels: male female

 Note that you supply the desired reference category as a character string to
 the ref argument, and you must take care to ensure its value matches exactly
 (including case sensitivity) one of the existing levels of the target factor.
      You can see that relevel changes only the category that comes first by
 operating on a factor with more than two levels. Using mob.fac from earlier,
 consider the following:

 R> relevel(x=mob.fac,ref="Dec")
 [1] Apr Jan Dec Sep Nov Jul Jul Jun
 Levels: Dec Jan Feb Mar Apr May Jun Jul Aug Sep Oct Nov

 Here, you ask that December becomes the first category; all other levels
 remain in the same order as originally defined when mob.fac was created.
 (Note also that you haven’t actually changed mob.fac or sex.char.fac as they’re
 stored; to do that, you need to overwrite/reassign the objects with their
 releveled selves.) The reference level or “first” category of a factor affects
 the treatment of categorical variables in some situations (you’ll see examples
 of this in Chapter 21’s discussion of multiple regression models).

                  4.3.3 Dropping Categories
                  Remember that changes to the levels of a factor, both in terms of the avail­
                  able categories and in terms of their ordering, are far from just cosmetic.
                  There’s a big difference, for example, between a dataset with zero obser­
                  vations in some of the categories and the same dataset defined with fewer
                  categories to begin with. That said, you may occasionally need to simplify a
                  factor by removing categories that aren’t represented by the observation set
                  at hand. This typically occurs after you’ve subsetted a larger dataset and are
                  interested only in populated categories.
                      Consider again the mob.fac object defined in the previous section. You
                  created this specifically to include levels that weren’t represented by the
                  entries in the mob vector; that is, they were associated with “zero counts”
                  when you tabulated the factor. The object is reprinted here:

                  R> mob.fac
                  [1] Apr Jan Dec Sep Nov Jul Jul Jun
                  Levels: Jan Feb Mar Apr May Jun Jul Aug Sep Oct Nov Dec

                      Suppose you’re given this factor but are interested only in analyzing
                  the categories represented by the data. The droplevels function creates a
                  new factor with the exact same data but strips away any categories with zero
                  counts:

                  R> mob.fac2
                  [1] Apr Jan Dec Sep Nov Jul Jul Jun
                  Levels: Jan Apr Jun Jul Sep Nov Dec

                  Notice that the remaining categories of the new mob.fac2 are precisely the
                  same as those of the mob.fac.basic vector created at the very beginning of
                  Section 4.3.2, but their order is different (retaining the order imposed by
                  mob.fac).

 and, for one reason or another, you wish to collapse the factor to have fewer

 together these rarer levels into some overall “Other” category.
     To illustrate this, we’ll use the mob.fac object created in Section 4.3.2 on
 page 84, detailing the month of birth of eight individuals (Table 4­4). As a

                       Suppose we wish to collapse this object so that the unique categories
                  are not the individual months, but instead are coded as quarters. That is,
                  we wish to create a new object whereby all individuals with a birth month
                  of January, February, or March are assigned the same category, Q1; those
                  with a birth month of either April, May, or June are categorized as Q2, and
                  so on. One way to do this would be to convert the factor to its numeric rep­
                  resentation (as per Section 4.3.4), recognize the fact that numbers 1 through
                  3 would correspond to Q1, 4 through 6 would be Q2, and so on, and use
                  this to construct a new factor from scratch with the factor function. This is
                  a lot of work! Instead, we can use the existing factor and simply re­define

                  for collecting together multiple objects. You’ve not yet met lists; these will be
                  scrutinized in the following chapter (Section 5.1).

                  copy of mob.fac (named mob.fac.q) and subsequently proceeds to manipulate

                  R> mob.fac.q <- mob.fac
                  R> levels(x=mob.fac.q) <- list(Q1=c("Jan","Feb","Mar"),Q2=c("Apr","May","Jun"),

                  Note that the key to this action involves overwriting (or assigning) the levels
                  attribute of the target factor object with the desired definition. In a call to
                  list, you collect the labels of the pre­existing factor that you want to collapse
                  together as a vector of character strings, and assign each vector to a tag that
                  will become the new label (the latter needn’t be supplied in double quotes).

                  made up of four unique levels, labeled Q1, Q2, Q3, and Q4. The new category Q1
                  will be given to any of the original observations that correspond to January,
                  February, or March, while Q2 will represent any entries corresponding to
                  April, May, or June, and so on. You must ensure the character string labels
                  you’re collecting together match the levels of the original factor exactly.

 4.3.6 Combining Factors
 As you’ve seen, it’s usually simple to combine multiple vectors of the same
 kind (whether numeric, logical, or character) using the c function. In R ver­
 sions ≥ 4.1.0, you can do the same thing. As a quick toy example, consider
 the following two factor objects, each of which has four elements:

 R> foo <- factor(x=c("A","B","A","C"))
 R> foo
 [1] A B A C
 Levels: A B C
 R> bar <- factor(x=c("D","E","E","E"))
 R> bar
 [1] D E E E
 Levels: D E

 Note that foo and bar are completely different objects; the former has three
 levels labeled "A", "B" and "C", and the latter has only two possible categories,
 "D" and "E". Create a new factor by combining everything via c:

 R> baz <- c(foo,bar)
 R> baz
 [1] A B A C D E E E
 Levels: A B C D E

 Here, the new factor has eight observations spanning five possible categories.
     More realistically, you may simply want to add newly obtained observa­
 tions to an existing object, where the new observations take on categories
 the existing object has already defined. Let’s see this behavior on the data
 in Table 4­4; specifically, the MOB factor vector mob.fac, from Section 4.3.2.
 Suppose you now observe three more individuals with MOB values "Oct",
 "Feb", and "Feb". You can quickly create something like

 R> new.mob.values <- factor(x=c("Oct","Feb","Feb"))
 R> new.mob.values
 [1] Oct Feb Feb
 Levels: Feb Oct

 and append them to the existing observations of mob.fac:

 R> mob.fac.ext <- c(mob.fac,new.mob.values)
 R> mob.fac.ext
 [1] Apr Jan Dec Sep Nov Jul Jul Jun Oct Feb Feb
 Levels: Jan Feb Mar Apr May Jun Jul Aug Sep Oct Nov Dec

 The three new entries now appear at the end of the eight original values.
 Furthermore, the fact that the levels of new.mob.values (only “Feb” and “Oct”)
 aren’t exactly equivalent to the entire set of 12 possible levels of mob.fac
 doesn’t matter; R recognizes that Feb and Oct already exist as category labels
 in mob.fac.

                  4.3.7 Creating Factors from Continuous Values
                  You often create factors from data originally measured on a continuum, for
                  example the weight of a set of adults or the amount of a drug given to a
                  patient. Sometimes you need to group (or bin) these types of observations
                  into categories, like Small/Medium/Large or Low/High. In R, you can
                  mold these kinds of data into discrete factor categories using the cut func­
                  tion. Consider the following numeric vector of length 10:

                  R> Y <- c(0.53,5.4,1.5,3.33,0.45,0.01,2,4.2,1.99,1.01)

                      Suppose you want to bin the data as follows: Small refers to observa­
                  tions in the interval [0, 2), Medium refers to [2, 4), and Large refers to [4, 6].
                  A square bracket refers to inclusion of its nearest value, and a parenthesis
                  indicates exclusion, so an observation y falls in the Small interval if 0 ≤ y < 2,
                  in Medium if 2 ≤ y < 4, or in Large if 4 ≤ y ≤ 6. Use cut and supply your
                  desired break intervals to the breaks argument:

                  R> br <- c(0,2,4,6)
                  R> cut(x=Y,breaks=br)
                  [1] (0,2] (4,6] (0,2] (2,4] (0,2] (0,2] (0,2] (4,6] (0,2] (0,2]
                  Levels: (0,2] (2,4] (4,6]

                      This gives you a factor, with each observation now assigned an interval.
                  However, notice that the boundary intervals are back­to­front; you want the
                  boundary levels on the left like [0,2), rather than the right as they appear by
                  default, (0,2]. Fix this by setting the logical argument right to FALSE:

                  R> cut(x=Y,breaks=br,right=FALSE)
                  [1] [0,2) [4,6) [0,2) [2,4) [0,2) [0,2) [2,4) [4,6) [0,2) [0,2)
                  Levels: [0,2) [2,4) [4,6)

                       Now you’ve swapped which boundaries are inclusive and exclusive. This
                  is important because it changes which categories the values fall into. Notice
                  that the seventh observation has changed categories. But there’s still a prob­
                  lem: The final interval currently excludes 6, and you want this maximum
                  value to be included in the highest level. You can fix this with another logical

90   Chapter 4
Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

 argument, include.lowest, which you also use to include the highest value if
 right is FALSE, as the ?cut help file indicates:

 R> cut(x=Y,breaks=br,right=FALSE,include.lowest=TRUE)
 [1] [0,2) [4,6] [0,2) [2,4) [0,2) [0,2) [2,4) [4,6] [0,2) [0,2)
 Levels: [0,2) [2,4) [4,6]

      The intervals are now defined how you want. Finally, you can add bet­
 ter labels to the categories by passing a character string vector to the labels
 argument, rather than using the interval levels that R applies by default:

 R> lab <- c("Small","Medium","Large")
 R> cut(x=Y,breaks=br,right=FALSE,include.lowest=TRUE,labels=lab)
 [1] Small Large Small Medium Small Small Medium Large Small Small
 Levels: Small Medium Large

 The order of your supplied labels must match the order of the levels in the
 factor object.

 4.4.1 Creating and Formatting Dates
 Let’s start simple. You can write a date down in many different ways and
 store it as a character string. For example, the following objects d1, d2, and d3
 all represent the same date:

 R> d1 <- "2025-02-19"
 R> d2 <- "2/19/25"
 R> d3 <- "19Feb2025"

 As character strings, you can’t perform any operations with these objects
 beyond using them as names or labels. To instruct R to interpret them as
 proper calendar dates, use the as.Date function and, in so doing, ensure R
 knows precisely how you’ve formatted the string so it correctly identifies the
 year, month, and day. The default is a “year­month­day” format; specifically:

                      YYYY­MM­DD            or       YYYY/MM/DD

      In the strings, only d1 is in one of these specific forms. We tell R to treat
 it as a proper date with the following call:

 R> date1 <- as.Date(x=d1)
 R> date1
 [1] "2025-02-19"

 At a glance, it seems nothing has changed; date1 looks just like the charac­
 ter string d1 from which it was created. But internally, the class of the object
 has changed, meaning you can ultimately do things with it that make sense
 in terms of calendar dates (you’ll look at more technical details related to
 object classes in Section 6.2.2).

 4.4.2 Performing Simple Operations with Dates
 Now that you’ve learned how to create formal objects for dates, let’s explore
 how they differ from basic character strings, using the d1 and date1 objects
 created in Section 4.4.1. Recall from Section 4.2.1 that attempting, say, an
 arithmetic operation on a character string rightly results in an error:

 R> d1
 [1] "2025-02-19"
 R> d1 + 1
 Error in d1 + 1 : non-numeric argument to binary operator

 But look what happens if you instead try to “add 1” to the formal date object
 of d1, namely date1:

 R> date1
 [1] "2025-02-19"

                                                                     Non-Numeric Values   95
                 Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

                  R> date1 + 1
                  [1] "2025-02-20"

                  Here, the operation returns a new date in which the day has been incre­
                  mented by one.
                       The preceding example hints to the underlying numeric representation
                  of a date object. Consider the result of the as.numeric function directly on
                  date1:

                  R> as.numeric(date1)
                  [1] 20138

                  The resulting number expresses the given date in terms of the number of
                  days since January 1, 1970 (known as the “Unix epoch”). This is clear if you
                  look at the numeric form of this particular date:

                  R> as.numeric(as.Date(x="1970-01-01"))
                  [1] 0

                      Dates before the Unix epoch have a corresponding negative number; for
                  example, the following line shows Christmas Day in 1969 as being seven days
                  prior to this point:

                  R> as.numeric(as.Date(x="1969-12-25"))
                  [1] -7

                       The utility of the date format in this sense now starts to become appar­
                  ent. If your desired tasks depend upon measuring time from some fixed
                  point, then you needn’t explicitly concern yourself with the differing days
                  of varying months or leap years; R handles all of that based on the dates
                  of interest. For example, although date1 + 1 is trivial, suppose you need to
                  calculate the date exactly 139 days from February 19, 2025. You can find this
                  as July 8 of the same year:

                  R> date1 + 139
                  [1] "2025-07-08"

                      In much the same fashion, it’s also easy to see how many days separate
                  two dates:

                  R> term <- as.Date(x="1947-07-30")
                  R> ramb <- as.Date(x="1946-07-06")
                  R> term - ramb

 4.4.3 Incorporating Time
 The principles from the previous section carry through to full date­time def­
 initions, where you can additionally specify hours, minutes, seconds, and
 even fractions of seconds. Unsurprisingly, this comes with additional com­
 plexity. Definitions of and calculations involving such objects must respect
 time zones, which in turn can raise even more issues, such as handling day­
 light savings. For these reasons we’ll look at a couple of simple examples here.
      First, consider the following command, which, when you execute it at
 the R prompt, gives the present date and time as reported by your local
 machine (your result will be different to that displayed here!):

 R> Sys.time()
 [1] "2022-03-14 14:15:08 NZDT"

 I wrote this sentence shortly after a quarter past two in the afternoon on
 New Zealand daylight saving time, March 14, 2022. The output hints at the
 standard form for full date­time specifications; following the date itself, the
 time is given as two­digit 24­hour, minute, and second components sepa­
 rated by colons, and there is clearly an allowance for time zone.




                                                                     Non-Numeric Values   97
                 Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies

                      Given a character string provided in this way, the strptime function can
                  create the corresponding date­time object. For example, you can create
                  34 minutes and 12 seconds past 7 PM on October 31, 2022 as follows:

                  R> dtime1 <- strptime(x="2022-10-31 19:34:12",format="%Y-%m-%d %H:%M:%S")
                  R> dtime1
                  [1] "2022-10-31 19:34:12 NZDT"

                  Here, note two things: First, you use strptime in the same way as as.Date,
                  whereby the first argument is a string made up of the desired date­time com­
                  ponents, and the second uses recognized conversion specifications to define
                  the format (see Table 4­6).

                  Table 4-6: Common Time-Specific Conversion
                  Specifications for Date-Time Objects
                   Time marker     Role
                   %H              Hours (in 2-digit, 24-hour form)
                   %M              Minutes (as 2-digit numeric)
                   %S              Seconds (as 2-digit numeric)


                      Second, the created object has been assigned a time zone; by default, R
                  queries your system locale. In this case, it’s correctly identified I’m in New
                  Zealand, and the specified date falls during daylight saving time (NZDT).
                      You can dictate specific time zones using the optional tz argument in
                  strptime. You can instruct R to store the same date­time as Greenwich Mean
                  Time / UTC by supplying the string "GMT":

                  R> dtime2 <- strptime(x="2022-10-31 19:34:12",format="%Y-%m-%d %H:%M:%S",
                                        tz="GMT")
                  R> dtime2
                  [1] "2022-10-31 19:34:12 GMT"

                  By typing OlsonNames() at your R prompt, you’re provided the collection of
                  time zone values able to be supplied to tz in a command like strptime. Query
                  what R identifies your present time zone to be by typing Sys.timezone() at the
                  prompt.
                       As best it can (there are various subtleties that can impart discrepan­
                  cies between systems), R automatically identifies daylight savings statuses
                  based on the provided date. For example, using the OlsonNames() entry for
                  US Pacific time, note that the same date­time yields a PDT suffix, denoting
                  Pacific daylight saving:




98   Chapter 4
       Book of R, 2nd edition (Sample Chapter) © 09/05/2025 by Tilman M. Davies


        R> dtime3 <- strptime(x="2022-10-31 19:34:12",format="%Y-%m-%d %H:%M:%S",
                              tz="US/Pacific")
        R> dtime3
        [1] "2022-10-31 19:34:12 PDT"

        However, if you set a date­time outside the usual daylight saving part of the
        year using the same time zone, like Christmas Day, R identifies the resulting
        object as Pacific Standard Time:

        R> dtime4 <- strptime(x="2022-12-25 08:45:00",format="%Y-%m-%d %H:%M:%S",
                              tz="US/Pacific")
        R> dtime4
        [1] "2022-12-25 08:45:00 PST"

            As always, these seemingly minor technicalities aren’t just cosmetic;
        arguably the most important consequences arise when using the date­time
        objects in calculations. Taking the difference between dtime1 and dtime2
        (which are defined to be exactly the same date­time, just in different time
        zones), for example, tells you that in New Zealand daylight saving time is
        13 hours ahead of GMT:

        R> dtime1 - dtime2
        Time difference of -13 hours

        Despite the order of differencing, the result is given as a negative here, imply­
        ing that the New Zealand Daylight Time version, dtime1, needs 13 hours added
        to it in order for there to be zero difference between the two zones.

NOTE    Remember, daylight saving times are introduced, removed, or changed often at the
        whim of governments. As such, it’s quite possible that the exact results of the preced­
        ing examples may well change when you run them in the future!
             Finally, as earlier, note there is an underlying numeric representation of
        a full date­time object. In Section 4.4.2 this was provided in days since the
        Unix epoch (January 1, 1970) for date­only values. For full date­times, the
        default use of as.numeric function provides the converted value in terms of
        the number of seconds since January 1, 1970, at 00:00:00 GMT:

        R> as.numeric(dtime1)
        [1] 1667198052
